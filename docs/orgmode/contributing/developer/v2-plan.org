* Current State
~nwchemc~ exposes a C ABI that accepts the Cap'n Proto ~NWChemParams~ message
directly. The public calculation entry points are:

- ~nwchemc_set_params~
- ~nwchemc_energy~
- ~nwchemc_energy_gradient~
- ~nwchemc_energy_forces~
- ~nwchemc_hessian~
- ~nwchemc_session_create~
- ~nwchemc_session_set_params~
- ~nwchemc_session_destroy~
- ~nwchemc_session_energy~
- ~nwchemc_session_energy_gradient~
- ~nwchemc_session_energy_forces~
- ~nwchemc_session_calculate_forces~
- ~nwchemc_session_calculate_result~
- ~nwchemc_calculate_result~
- ~nwchemc_session_calculate_hessian~
- ~nwchemc_session_hessian~
- ~nwchemc_potential_result_size_for_force_input~
- ~nwchemc_available~
- ~nwchemc_version~
- ~nwchemc_finalize~

The parser uses ~capnp-c~ and keeps Cap'n Proto as the language-neutral
configuration carrier. The runtime does not copy the parameter tree into a
second public configuration struct.
Long-running callers can keep one ~NWChemParams~ configuration in a session and
send per-step ~ForceInput~ Cap'n Proto messages for coordinates, atomic
numbers, length units, optional cell vectors, and unit-aware
~PotentialResult~ output.

* RTDB Path
The embedded Fortran layer writes numeric geometry through NWChem geometry APIs
and writes Gaussian library basis data through NWChem basis APIs. Session
~ForceInput.box~ values are applied as numeric 3x3 cells through
~geom_amatrix_set~, ~geom_isystype_set(3)~, and ~geom_P1_set~. The bridge then
stores charge, multiplicity, method, and task state through RTDB-facing calls
before calling ~task_energy~, ~task_gradient~, or ~task_hessian~.

NWChem modules read and mutate RTDB entries inside task routines and module
input handlers. Direct RTDB writes must therefore preserve this order:

1. Initialize GA/MA/RTDB and file-directory state.
2. Store numeric geometry and basis records.
3. Apply caller-provided structured or raw NWChem stanzas.
4. Store scalar method/task keys.
5. Call the NWChem task routine.
6. Read task output from RTDB or the NWChem task output file.

~nw_inp_from_character~ remains reserved for caller-provided NWChem stanzas.
Geometry coordinates and library basis selection are not routed through text
input. Typed SCF scalar controls (~maxiter~, ~thresh~, and ~tol2e~), driver
~maxiter~, driver convergence presets (~tight~ / ~loose~), and explicit driver
~gmax~ / ~grms~ / ~xmax~ / ~xrms~ tolerances are promoted into direct RTDB
writes for embed calculations and omitted from embed-rendered text when no
non-promoted syntax remains.

* Structured Coverage
The schema contains typed stanzas for:

- generic module blocks
- ~dft~ blocks including ~xc~, ~direct~, and Fermi smearing syntax
- ~scf~ blocks including typed max-iteration, convergence-threshold, and
  two-electron tolerance scalars
- ~driver~ blocks including direct max-iteration promotion and text fallback
  for other optimization controls
- ~set~ directives
- raw NWChem fragments
- module blocks backed by the ~NWChemModuleName~ enum
- ~nwpw~ pseudopotential blocks with library, PSPW, PAW, CPI, and Teter entries
  plus structured ~allElements~ default entries
- per-step ~ForceInput~ messages with atoms, coordinates, length units, and
  optional row-major 3x3 cells
- the ~PotentialService.calculate~ RPC shape around ~ForceInput~ and
  ~PotentialResult~

The feature inventory in ~schema/inventory/nwchem_features.json~ interns every
schema field, stanza kind, module enum value, and ABI entry point. The cmocka
feature tests verify the generated inventory lookup table and ABI coverage.

* Pickup Tasks
** Long-Running Step Calls
~nwchemc_session_calculate_forces~,
~nwchemc_session_calculate_result~, and
~nwchemc_session_calculate_hessian~ are C ABI entry points for socket-style,
optimizer-style, and vibrational callers. They take the session's persistent
~NWChemParams~ and a per-call serialized ~ForceInput~ message, validate atom
counts and output buffer sizes, convert coordinates and cells to the C ABI
units, call the embed bridge, and return either native NWChem buffers or a
serialized ~PotentialResult~ in the caller's requested units.
~nwchemc_potential_result_size_for_force_input~ gives callers the
~PotentialResult~ byte count from the per-step ~ForceInput~ without evaluating
NWChem.
~nwchemc_calculate_result~ is the one-shot convenience wrapper around the same
~NWChemParams + ForceInput -> PotentialResult~ path. Multi-step callers should
reuse a session so configuration is applied once and per-step calls only carry
coordinates, cell data, atom numbers, and requested units.

The session path writes geometry and cell data numerically. Maintain that shape
for coordinates, atom numbers, and cell vectors. Text input is acceptable only
for NWChem grammar that does not have a confirmed direct API or RTDB key.
Text-valued ~NWChemSetDirective~ stanzas are already promoted to direct
~rtdb_cput~ calls in the embed path, so keep simple RTDB string options out of
~nw_inp_from_character~.

The result-carrier path keeps the native C force buffer ABI available for
low-overhead callers and gives RPC-style callers the same ~PotentialResult~
message used by rgpot.

** DONE (partial) Direct DFT promotion
~nwchemc_params_extract_direct_dft~ pulls ~xc~, ~direct~, and smearing from the
typed ~NWChemDftStanza~ values. ~apply_config_to_embed~ uses
~nwchemc_params_render_embed_input_blocks~ (omits promoted DFT text) and
~nwchemc_embed_set_dft_direct~ to apply xc via ~cfg_scf~/~dft:xc~. The
fixed-form RTDB setup writes ~dft:direct~, ~int2e:*~, ~dft:smear_sigma~, and
~dft:spinset~ for gradient and Hessian paths. Parser test:
~test_parser_extracts_direct_dft_options~ in ~tests/test_capnp_parser.c~;
embed-boundary test: ~tests/cmocka/test_embed_config_cmocka.c~.

~nwchemc_params_extract_direct_scf~ pulls typed ~NWChemScfStanza.maxiter~,
~thresh~, and ~tol2e~ values into ~nwchemc_embed_set_scf_direct~. The fixed-form
bridge writes ~scf:maxiter~, ~scf:thresh~, and ~scf:tol2e~ for SCF, and
~dft:iterations~ / ~dft:tol2e~ where the DFT handler has matching keys.

~nwchemc_params_extract_direct_set_strings~ pulls text-valued
~NWChemSetDirective~ entries into ~nwchemc_embed_set_rtdb_strings~. The
fixed-form bridge writes those entries with ~rtdb_cput~ for gradient and
Hessian paths before method state is applied.

~nwchemc_params_extract_direct_driver~ pulls typed
~NWChemDriverStanza.maxiter~, ~tight~, ~loose~, ~gmaxTol~, ~grmsTol~,
~xmaxTol~, and ~xrmsTol~ into ~nwchemc_embed_set_driver_direct~. The
fixed-form bridge writes ~driver:nptopt~ and the four convergence tolerance keys
(~driver:gmax_tol~, ~driver:grms_tol~, ~driver:xrms_tol~,
~driver:xmax_tol~), matching ~driver_input.F~ and ~opt_drv.F~. Explicit scalar
values override preset defaults before the RTDB writes.

Open direct-path checks: migrate task theory/operation overrides only where
RTDB keys are confirmed; add embed regression that asserts direct path matches
text-path energies when NWChem embed is available.

** Expand Direct RTDB Settings
Move additional scalar options that have stable NWChem RTDB keys out of rendered
text stanzas. Good candidates are task operation/theory overrides beyond embed
defaults and any additional module scalar controls with confirmed RTDB keys.

Each migrated option needs a cmocka parser test, a real NWChem test that proves
the direct path still computes finite outputs, and a reference to the upstream
NWChem input routine or RTDB key being matched.

** Pseudopotential Direct Path
Keep the structured pseudopotential Cap'n Proto stanza. Replace the rendered
~nwpw~ pseudopotential block only where NWChem exposes a direct API or stable
RTDB representation that exactly matches the input handler. Pseudopotentials are
used by PSPW/NWPW and related rgpot potential workflows, so preserve library
type semantics instead of collapsing them to free-form strings.

** Atom And Geometry Limits
The fixed-form embed helper currently caps direct geometry and basis setup at 64
atoms. Increase this only by matching NWChem geometry/basis allocation behavior,
not by widening local arrays alone. Atomic-number to element-symbol mapping uses
the full periodic table through oganesson for direct geometry and basis setup.

** ABI Surface
Keep adding calculation entry points around the same ~NWChemParams~ carrier:
energy-only, properties, dipoles, optimized geometries, vibrational analysis,
and any rgpot-relevant potential/force forms. Avoid parallel option structs.

** Build Model
The embed build consumes an NWChem build tree because it needs source headers,
private module headers, static archives, ~stubs.o~, and GA config metadata. A
conda-forge runtime package is useful only if it exposes equivalent embed
artifacts and link metadata.
